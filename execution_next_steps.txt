Execution Layer V2 — WebSocket-Enhanced Order Tracking
======================================================

Goal: Add real-time fill/order tracking via WebSockets on both platforms.
Order submission remains REST — neither platform supports WS order placement.

Current flow:
  User → BUY KS YES 10 → REST POST → HTTP response → done (no further tracking)

Target flow:
  User → BUY KS YES 10 → REST POST → HTTP response (order accepted)
                                     ↓
                          WS fill channel → real-time fill notification
                          WS order channel → resting/canceled/executed updates
                                     ↓
                          Frontend → toast, order status, fill history


================================================================================

================================================================================

No new connection needed. Our existing authenticated KS WebSocket can subscribe
to private channels on the same connection.

Tasks:

================================================================================
STEP 5: Order State Management & Edge Cases
================================================================================

Tasks:
  [ ] Correlate REST order_id with WS fill events:
      - Kalshi: REST response contains order_id, fill msg has order_id
      - Polymarket: REST response contains order ID, trade event has market + side

  [ ] Handle partial fills:
      - KS user_order has fill_count_fp + remaining_count_fp
      - PM order UPDATE has size_matched vs original_size
      - Show "5/10 filled" in UI

  [ ] Handle FOK (fill-or-kill) for PM market orders:
      - FOK either fills completely or fails immediately
      - REST response is sufficient — WS is confirmation only
      - No resting state possible

  [ ] Handle GTC (good-till-canceled) for PM limit orders:
      - Order may rest indefinitely
      - WS user channel is the only way to know when it fills
      - Show "resting" status in UI until fill/cancel

  [ ] Handle order expiry:
      - KS: orders may have expiration_time
      - PM: GTC orders persist until filled or canceled

  [ ] Reconnection resilience:
      - If KS WS reconnects, re-subscribe to fill + user_orders
      - If PM user WS reconnects, re-subscribe with auth + current markets
      - Orders placed during disconnection: rely on REST response,
        pick up WS tracking on reconnect

  [ ] Cleanup: remove orders from activeOrders after terminal state reached
      + configurable retention for fillHistory
