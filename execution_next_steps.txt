Execution Layer V2 — WebSocket-Enhanced Order Tracking
======================================================

Goal: Add real-time fill/order tracking via WebSockets on both platforms.
Order submission is REST currently; Kalshi FIX API is the future target (Step 6).

Current flow:
  User → BUY KS YES 10 → REST POST → HTTP response → done (no further tracking)

Target flow:
  User → BUY KS YES 10 → REST POST → HTTP response (order accepted)
                                     ↓
                          WS fill channel → real-time fill notification
                          WS order channel → resting/canceled/executed updates
                                     ↓
                          Frontend → toast, order status, fill history

STEP 5B: Polymarket REST Heartbeat (CRITICAL for GTC/GTD orders)
================================================================================

From Polymarket docs (trading/orders/overview):
  "If a valid heartbeat is not received within 10 seconds (with up to a
   5-second buffer), all of your open orders will be cancelled."

This is a REST API heartbeat, NOT the WebSocket PING. It only matters
when we have resting orders (GTC or GTD). FOK market orders fill or
fail instantly so they don't need heartbeat.

If we ever place GTC limit orders on Polymarket, we MUST implement this
or all resting orders will be auto-canceled after ~15 seconds.

Tasks:
  [ ] Investigate heartbeat endpoint:
      - Likely a REST POST with a heartbeat ID returned by the server
      - Each heartbeat response includes the next heartbeat ID to send
      - Must send every 5 seconds (10s timeout with 5s buffer)

  [ ] Implement heartbeat loop in api_server or btc_watcher:
      - Start when a PM GTC/GTD order is placed
      - Stop when no resting PM orders remain
      - If heartbeat fails, warn user that orders may be canceled

  [ ] Alternative: use FOK/FAK for all PM orders (no heartbeat needed)
      - FOK: fill completely or cancel entirely
      - FAK: fill what's available, cancel remainder
      - Simpler but no resting order support

Files: clients/executor.py, api_server.py

Note: For now, we only use FOK market orders on PM, so heartbeat is not
yet required. This becomes critical when we add GTC limit order support.

  [ ] Reconnection resilience:

STEP 6: Kalshi FIX API — Order Placement via FIX Protocol
================================================================================

Goal: Replace REST order placement with Kalshi's FIX (Financial Information
eXchange) protocol for lower latency and richer order management.

Background:
  FIX is the industry-standard protocol used by institutional trading systems.
  Kalshi offers a FIX API that supports order entry, modification, cancellation,
  RFQ, and drop copy — all over a persistent session instead of per-request HTTP.
  Docs: https://docs.kalshi.com/fix

Advantages over REST:
  - Persistent session — no per-request TCP/TLS + RSA signing overhead
  - Submit, modify, and cancel orders through standard FIX messages
  - Real-time execution reports on the same session (no separate WS needed)
  - Drop copy session for order event recovery (replay missed fills)
  - RFQ support for market making
  - Order groups with automatic cancellation and contract limits
  - Industry standard — compatible with existing FIX libraries/engines

Prerequisites:
  - Contact institutional@kalshi.com for FIX API access
  - Generate 2048-bit RSA PKCS#8 key pair
  - Upload public key to receive FIX API key
  - Download Kalshi FIX Dictionary (XML) for message specs

Tasks:
  [ ] Obtain FIX API access from Kalshi (institutional@kalshi.com)

  [ ] Evaluate Python FIX libraries:
      - quickfix (most established, C++ engine with Python bindings)
      - simplefix (lightweight, pure Python)
      - Custom implementation if neither fits

  [ ] Implement FIX session manager:
      - Logon with RSA key authentication
      - Heartbeat / session keep-alive
      - Logout and reconnection handling
      - Sequence number management

  [ ] Implement FIX order entry:
      - NewOrderSingle (35=D) for order placement
      - OrderCancelRequest (35=F) for cancellation
      - OrderCancelReplaceRequest (35=G) for modification
      - Parse ExecutionReport (35=8) for fill/status updates

  [ ] Implement drop copy session:
      - Separate FIX session for order event recovery
      - Replay missed fills on reconnection

  [ ] Migrate executor.py:
      - Replace place_kalshi_order() REST call with FIX NewOrderSingle
      - FIX execution reports replace WS fill/user_orders channels
      - Keep REST as fallback if FIX session is down

  [ ] Update api_server.py:
      - Route FIX execution reports to /ws/trade broadcast
      - Correlate with _active_orders tracking

Files:
  clients/kalshi_fix.py (new) — FIX session manager + order entry
  clients/executor.py — switch Kalshi order path to FIX
  api_server.py — route FIX execution reports to frontend

Note: This replaces both REST order placement AND WS fill/order tracking
for Kalshi with a single FIX session. The WS connection would still be
used for market data (ticker, orderbook_delta) unless Kalshi also offers
market data over FIX.

*** IMPORTANT: Refactor to Independent Data Streams ***
================================================================================

Current architecture bottleneck: ALL data sources (BRTI, Kalshi orderbook,
PM orderbook, Chainlink) funnel through a single _push_update() that
rebuilds one monolithic snapshot and sends it over one WebSocket message.

Problems:
  1. All-or-nothing push — a tiny Kalshi bid change rebuilds the entire
     snapshot including BRTI, PM data, Chainlink, staleness checks
  2. Sources block each other — 0.1s throttle means a BRTI update can
     be silently dropped if a Kalshi orderbook delta just fired
  3. No priority — a fill notification (critical) is throttled the same
     as a routine Chainlink price tick
  4. Redundant data — every push sends ~2KB of unchanged Kalshi/PM data
     even if only the BRTI price changed

Target architecture (Bloomberg-style independent streams):
  BRTI price ──→ {"type": "brti",      "price": 68142.35}      ──→ frontend
  KS orderbook ──→ {"type": "ks_book",  "yes_bid": 0.52, ...}  ──→ frontend
  PM orderbook ──→ {"type": "pm_book",  "up_ask": 0.49, ...}   ──→ frontend
  Chainlink ──→ {"type": "chainlink",   "price": 68140.20}     ──→ frontend
  Fills/orders ──→ {"type": "ks_fill",  ...}                    ──→ frontend (no throttle)

Benefits:
  - No source can starve another
  - Minimal data over the wire (only changed fields)
  - Each stream has its own update rate
  - Critical events (fills, order confirms) bypass price throttle
  - Frontend merges into Zustand store independently

Tasks:
  [ ] Split btc_update into granular message types on /ws/btc
  [ ] Update frontend onBtcMessage to handle multiple message types
  [ ] Remove monolithic snapshot rebuild — each source pushes its own delta
  [ ] Critical events (fills, order status) bypass throttle entirely
  [ ] Each data source manages its own push interval

Files: api_server.py, clients/btc_watcher.py, terminal/src/App.tsx,
       terminal/src/store.ts, terminal/src/types.ts
