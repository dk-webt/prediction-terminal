Data Stream & Execution V2 — Implementation Plan
==================================================

Current state: WS connections are torn down and rebuilt on every 15-min roll.
Orders are fire-and-forget (no fill tracking). Per-request RSA signing is correct
but uses wrong salt length (MAX_LENGTH instead of DIGEST_LENGTH per Kalshi docs).

This plan addresses 5 areas:
  1. Dynamic subscriptions (no reconnect on roll)
  2. Kalshi fill/order tracking via private WS channels
  3. Polymarket fill/order tracking via user channel
  4. Live BTC reference price via Polymarket RTDS
  5. Bug fixes (RSA salt length, PM custom features)


================================================================================
PLATFORM API REFERENCE
================================================================================

── Kalshi WebSocket ───────────────────────────────────────────────────────────

URL:      wss://api.elections.kalshi.com/trade-api/ws/v2
Auth:     RSA-PSS signed headers at handshake (KALSHI-ACCESS-KEY, SIGNATURE, TIMESTAMP)
Signing:  timestamp_ms + "GET" + "/trade-api/ws/v2", RSA-PSS SHA-256, salt=DIGEST_LENGTH
Keepalive: Server sends protocol-level Ping (opcode 0x9) every 10s with body "heartbeat"
           Client responds with Pong (opcode 0xA, empty body) — handled automatically by
           the websockets library. Client may also initiate Pings.

Commands:
  subscribe            — Subscribe to channels for markets. Returns SIDs.
  unsubscribe          — Cancel subscriptions by SID.
  update_subscription  — Add/remove markets from existing subscriptions (no reconnect).
  list_subscriptions   — List all active subscriptions.

Public channels:  ticker, orderbook_delta, trade, market_lifecycle_v2,
                  multivariate_market_lifecycle, multivariate
Private channels: fill, user_orders, market_positions, communications, order_group_updates

Execution: REST POST /trade-api/v2/portfolio/orders (RSA-PSS signed per-request)


── Polymarket WebSockets ──────────────────────────────────────────────────────

4 separate endpoints:

1) Market channel (what we currently use for data)
   URL:       wss://ws-subscriptions-clob.polymarket.com/ws/market
   Auth:      None required
   Keepalive: Client must send text "PING" every 10s, server responds "PONG"
   Subscribe: {"assets_ids": [...], "type": "market", "custom_feature_enabled": true}
   Dynamic:   {"operation": "subscribe", "assets_ids": [...]}
              {"operation": "unsubscribe", "assets_ids": [...]}
              (No "type" field on subsequent messages. No server acknowledgment.)

   Event types:
     book              — Full orderbook snapshot (on subscribe + trades)
     price_change      — Order placed/canceled, includes best_bid/best_ask per asset
     last_trade_price  — Trade executed (price, size, side, fee_rate_bps)
     tick_size_change  — Tick size changes at price extremes (>0.96 or <0.04)
     best_bid_ask*     — Best bid/ask shift (requires custom_feature_enabled)
     new_market*       — New market created (requires custom_feature_enabled)
     market_resolved*  — Market resolved with winner (requires custom_feature_enabled)

2) User channel (for fill/order tracking)
   URL:       wss://ws-subscriptions-clob.polymarket.com/ws/user
   Auth:      API key/secret/passphrase in subscribe message
   Keepalive: Client must send text "PING" every 10s
   Subscribe: {
                "auth": {"apiKey": "...", "secret": "...", "passphrase": "..."},
                "type": "user",
                "markets": ["condition_id_1", ...]
              }
   NOTE: Uses condition IDs, NOT asset/token IDs (unlike market channel)

   Event types:
     trade — Fill notification
       Status flow: MATCHED → MINED → CONFIRMED (success) or FAILED
       Fields: status, side, size, price, outcome (YES/NO), market, maker_orders
     order — Order lifecycle
       Types: PLACEMENT, UPDATE (partial fill), CANCELLATION
       Fields: original_size, size_matched, price, side, outcome, asset_id, market

3) RTDS — Real-Time Data Socket
   URL:       wss://ws-live-data.polymarket.com
   Auth:      Optional (gamma_auth with wallet address for user-specific streams)
   Keepalive: Client must send PING every 5s
   Subscribe: {
                "action": "subscribe",
                "subscriptions": [{
                  "topic": "crypto_prices_chainlink",
                  "type": "price",
                  "filters": "btc/usd"
                }]
              }

   Crypto price topics:
     crypto_prices           — Binance feed (filters: "btcusdt", "ethusdt", etc.)
     crypto_prices_chainlink — Chainlink feed (filters: "btc/usd", "eth/usd", etc.)

   Also available:
     equity_prices — Pyth Network (AAPL, TSLA, SPY, NVDA, etc.)
                     Includes 2-min historical snapshot on subscribe
     comments      — Platform comments/reactions (may require auth)

   Message format: {"topic": "...", "type": "...", "timestamp": N, "payload": {...}}

4) Sports channel
   URL:       wss://sports-api.polymarket.com/ws
   Auth:      None
   Keepalive: Server sends "ping" every 5s, client must respond "pong" within 10s
   Events:    sport_result (scores, periods, status)
   (Not relevant to BTC watcher — noted for completeness)


Execution: py-clob-client
  Market orders:  client.create_market_order(MarketOrderArgs) → client.post_order(signed, FOK)
  Limit orders:   client.create_and_post_order(OrderArgs) — always GTC
  Options:        PartialCreateOrderOptions(tick_size="0.01", neg_risk=False)

1. DYNAMIC SUBSCRIPTIONS (eliminates reconnect on roll)
================================================================================

Goal: Keep WS connections alive across 15-min window rolls. Swap tickers/tokens
in-place instead of disconnecting and reconnecting.

── 1A. Kalshi: update_subscription ─────────────────────────────────────────────

Current behavior (btc_watcher.py):
  - _window_roll_loop sets _ks_reconnect event
  - _kalshi_ws_loop sees the event, drops the connection, reconnects with new auth
  - Full reconnect = new TCP + TLS handshake + RSA-PSS auth + subscribe

New behavior:
  - Track SIDs returned from initial subscribe response
  - On roll, send update_subscription to add new ticker, then delete old ticker
  - No reconnect needed

Implementation:
  a) Store SIDs from subscribe response:
     - After sending subscribe for ["ticker", "orderbook_delta"], parse server
       responses of type "subscribed" and store { channel_name: sid }
     - e.g. self._ks_sids = {"ticker": 1, "orderbook_delta": 2}

  b) New method: _kalshi_update_subscription(ws, old_ticker, new_ticker)
     - Send two update_subscription commands per SID:
       {
         "id": <next_id>,
         "cmd": "update_subscription",
         "params": {
           "sids": [sid],
           "market_tickers": [new_ticker],
           "action": "add_markets",
           "send_initial_snapshot": true
         }
       }
       then:
       {
         "id": <next_id>,
         "cmd": "update_subscription",
         "params": {
           "sids": [sid],
           "market_tickers": [old_ticker],
           "action": "delete_markets"
         }
       }
     - Use list_subscriptions after swap to verify:
       {"id": N, "cmd": "list_subscriptions"}

  c) Change _window_roll_loop:
     - Instead of setting _ks_reconnect event, call _kalshi_update_subscription
       directly (need to pass ws reference or use a shared ref)
     - Store ws reference on self (self._ks_ws) so roll loop can send commands

  d) Change _kalshi_ws_loop:
     - Remove the roll_task / _ks_reconnect.wait() pattern
     - Keep only the recv loop + timeout/error handling
     - Reconnect only on actual connection failure

  e) Fallback: If update_subscription fails or server returns error, set a flag
     to trigger full reconnect (keep current reconnect logic as fallback)

Message format reference:
  Subscribe response:  {"id": 1, "type": "subscribed", "msg": {"channel": "ticker", "sid": 1}}
  Update sub:          {"id": N, "cmd": "update_subscription", "params": {"sids": [1], ...}}
  Unsubscribe:         {"id": N, "cmd": "unsubscribe", "params": {"sids": [1, 2]}}
  List subs:           {"id": N, "cmd": "list_subscriptions"}
  List response:       {"id": N, "type": "ok", "msg": [{"channel": "ticker", "sid": 1}, ...]}
  Error response:      {"id": N, "type": "error", "msg": {"code": N, "msg": "..."}}


── 1B. Polymarket: subscribe/unsubscribe on existing connection ────────────────

Current behavior:
  - _window_roll_loop sets _pm_reconnect event
  - _pm_ws_loop drops connection, reconnects, sends initial handshake with new tokens

New behavior:
  - On roll, send subscribe for new token IDs, then unsubscribe old ones
  - No reconnect needed

Implementation:
  a) Store ws reference: self._pm_ws

  b) Enable custom features on initial handshake:
     {"assets_ids": [...], "type": "market", "custom_feature_enabled": true}
     This gives us best_bid_ask, new_market, and market_resolved events.

  c) New method: _pm_swap_tokens(old_tokens, new_tokens)
     - Send on existing connection:
       {"operation": "subscribe", "assets_ids": new_tokens}
       {"operation": "unsubscribe", "assets_ids": old_tokens}
     - Note: server sends NO acknowledgment — success is silent

  d) Change _window_roll_loop:
     - Instead of setting _pm_reconnect, call _pm_swap_tokens
     - Update self._pm_token_ids after swap

  e) Change _pm_ws_loop:
     - Remove the roll_task / _pm_reconnect.wait() pattern
     - Keep recv + ping loops
     - Reconnect only on actual connection failure or inactivity timeout

  f) Fallback: If ws.send raises an exception during swap, trigger full reconnect

  g) Add new_market event handling in recv loop — could detect new contracts
     appearing without REST polling during rolls

Note: No "type" field on subsequent subscribe/unsubscribe messages.
      Only the initial handshake uses "type": "market".


── 1C. Roll loop changes ──────────────────────────────────────────────────────

Current _window_roll_loop flow:
  1. Sleep until boundary
  2. Clear PM data, set _rolling = True
  3. Retry loop: REST fetch both platforms until new contracts found
  4. Set _pm_reconnect / _ks_reconnect events
  5. Poll for PM strike price

New flow:
  1. Sleep until boundary
  2. Set _rolling = True (keep old data visible)
  3. Retry loop: REST fetch both platforms until new contracts found
  4. When PM ready: call _pm_swap_tokens(old_tokens, new_tokens)
  5. When KS ready: call _kalshi_update_subscription(ws, old_ticker, new_ticker)
  6. Set _rolling = False
  7. Poll for PM strike price

Benefits:
  - No TCP/TLS reconnect overhead (~200-500ms saved per roll)
  - No re-auth overhead for Kalshi
  - Smoother transition — old data keeps flowing until new subscription active
  - Eliminates "Task was destroyed but pending" warnings from Event.wait()

2. KALSHI FILL & ORDER TRACKING (private WS channels)
================================================================================

Goal: Get real-time notifications when orders are filled, canceled, or resting.
Currently we only know the result of the initial REST POST response.

── 2A. Subscribe to fill + user_orders channels ───────────────────────────────

These are private channels — require RSA-PSS auth at WS handshake time.
Our existing Kalshi WS connection already authenticates, so we can subscribe
on the same connection.

Implementation:
  a) After initial ticker/orderbook subscribe, also subscribe to:
     {
       "id": 3,
       "cmd": "subscribe",
       "params": { "channels": ["fill"] }
     }
     {
       "id": 4,
       "cmd": "subscribe",
       "params": { "channels": ["user_orders"] }
     }
     (Omit market_ticker to receive for all markets)

  b) Store SIDs for these channels too

  c) Handle new message types in _kalshi_recv_loop:
     - "fill": order was matched
       {
         "type": "fill",
         "sid": N,
         "msg": {
           "trade_id": "...",
           "order_id": "...",
           "market_ticker": "...",
           "is_taker": true,
           "side": "yes",
           "yes_price_dollars": "0.750",
           "count_fp": "278.00",
           "fee_cost": "...",
           "action": "buy",
           "ts": 1671899397,
           "post_position_fp": "500.00",
           "purchased_side": "yes"
         }
       }

     - "user_order": order lifecycle update
       {
         "type": "user_order",
         "sid": N,
         "msg": {
           "order_id": "...",
           "ticker": "...",
           "status": "resting|canceled|executed",
           "side": "yes|no",
           "yes_price_dollars": "0.7500",
           "fill_count_fp": "100.00",
           "remaining_count_fp": "0.00",
           "initial_count_fp": "100.00",
           "taker_fill_cost_dollars": "75.0000",
           "created_time": "...",
           "last_update_time": "..."
         }
       }

  d) New callback: self._on_fill(fill_data) and self._on_order_update(order_data)
     - Push to frontend via /ws/trade WebSocket
     - Frontend can show toast/notification for fills

  e) On roll, fill/user_orders subscriptions persist (they're not market-scoped
     when subscribed without market_ticker)

  f) Also available but lower priority:
     - market_positions: position updates
     - order_group_updates: grouped order status


── 2B. Frontend integration ───────────────────────────────────────────────────

  a) New message types on /ws/trade:
     - {"type": "ks_fill", "data": {...}}
     - {"type": "ks_order_update", "data": {...}}

  b) Store in Zustand: orderHistory, activeFills

  c) Show in UI:
     - Toast notification on fill
     - Order status indicator (resting → filled/canceled)
     - Fill history in detail panel (optional, lower priority)

3. POLYMARKET FILL & ORDER TRACKING (user channel)
================================================================================

Goal: Get real-time fill/order notifications for PM trades.

── 3A. User channel WebSocket ─────────────────────────────────────────────────

Separate authenticated WS endpoint:
  wss://ws-subscriptions-clob.polymarket.com/ws/user

Auth is via API creds in the subscribe message (NOT headers):
  {
    "auth": {
      "apiKey": "...",      // POLYMARKET_API_KEY from .env
      "secret": "...",      // POLYMARKET_API_SECRET from .env
      "passphrase": "..."   // POLYMARKET_API_PASSPHRASE from .env
    },
    "type": "user",
    "markets": ["condition_id"]   // NOTE: condition IDs, not token/asset IDs
  }

Implementation:
  a) New WS connection in BtcStreamManager: self._pm_user_ws
     - Runs alongside existing market channel connection
     - Separate ping loop (10s PING)
     - Needs condition_id (from market data, not token_id)

  b) Handle trade events:
     Status flow: MATCHED → MINED → CONFIRMED (success) or FAILED
     {
       "event_type": "trade",
       "status": "CONFIRMED",
       "side": "BUY",
       "size": "10",
       "price": "0.55",
       "outcome": "YES",
       "market": "condition_id",
       "maker_orders": [{"matched_amount": "10", "price": "0.55", "outcome": "Yes"}]
     }

  c) Handle order events:
     Types: PLACEMENT, UPDATE (partial fill), CANCELLATION
     {
       "event_type": "order",
       "type": "UPDATE",
       "original_size": "10",
       "size_matched": "5",
       "price": "0.55",
       "side": "BUY",
       "outcome": "YES"
     }

  d) Push to frontend via /ws/trade:
     - {"type": "pm_fill", "data": {...}}
     - {"type": "pm_order_update", "data": {...}}

  e) Priority note:
     - For market orders (FOK): REST response already confirms fill/reject,
       so user channel is a nice-to-have confirmation
     - For limit orders (GTC): user channel is essential — only way to know
       when a resting order gets filled
     - Implement after Kalshi fill tracking is working


── 3B. REST polling fallback (simpler alternative) ────────────────────────────

  If the user WS is too complex for now:
  - After placing a PM order, poll GET /orders?market=<condition_id> every few seconds
  - Check order status until filled/expired/canceled
  - Simpler but adds latency

4. LIVE BTC REFERENCE PRICE VIA RTDS
================================================================================

Goal: Show live BTC spot price alongside contract data so users can see how far
BTC has moved from strike during the 15-min window.

── 4A. Chainlink feed (matches PM strike source) ─────────────────────────────

The Polymarket RTDS streams the same Chainlink BTC/USD price that is used to
determine the strike price for BTC binary options. This is the most relevant
feed since it matches the resolution source.

Connection:
  URL: wss://ws-live-data.polymarket.com
  Auth: None required for crypto prices
  Keepalive: PING every 5s (stricter than market channel)

Subscribe:
  {
    "action": "subscribe",
    "subscriptions": [{
      "topic": "crypto_prices_chainlink",
      "type": "price",
      "filters": "btc/usd"
    }]
  }

Also available — Binance feed (higher frequency, different source):
  {
    "action": "subscribe",
    "subscriptions": [{
      "topic": "crypto_prices",
      "type": "price",
      "filters": "btcusdt"
    }]
  }

Messages arrive as:
  {"topic": "crypto_prices_chainlink", "type": "price", "timestamp": N, "payload": {...}}

── 4B. Implementation ─────────────────────────────────────────────────────────

  a) New WS connection in BtcStreamManager: self._rtds_ws
     - Connect to wss://ws-live-data.polymarket.com
     - Subscribe to crypto_prices_chainlink for btc/usd
     - Separate ping loop (5s PING — stricter requirement)

  b) Store live price: self._btc_spot_price
     - Update on each RTDS message
     - Include in push_update snapshot: {"btc_spot": 87234.56, "btc_spot_source": "chainlink"}

  c) Frontend display:
     - Show current BTC price in BTC watcher panel
     - Show delta from strike: "+$124.30 from strike" or "-$56.20 from strike"
     - Color green/red based on direction (up = green if holding UP, etc.)

  d) Optional: Also subscribe to Binance feed for comparison / faster updates
     - Could show both: "Chainlink: $87,234 | Binance: $87,238"

  e) This is independent of execution — can be implemented at any phase

5. BUG FIXES
================================================================================

── 5A. RSA-PSS salt length (both btc_watcher.py and executor.py) ──────────────

Current:  salt_length=padding.PSS.MAX_LENGTH
Docs say: salt_length=padding.PSS.DIGEST_LENGTH

Fix in both files:
  - clients/btc_watcher.py line 91: _kalshi_ws_auth_headers()
  - clients/executor.py line 73: _kalshi_rest_auth_headers()

Note: MAX_LENGTH works currently (Kalshi server accepts it), but DIGEST_LENGTH
is what the docs specify. Fix for correctness and forward compatibility.

── 5B. PM market channel: enable custom features ─────────────────────────────

Current subscribe:
  {"assets_ids": [...], "type": "market"}

Should be:
  {"assets_ids": [...], "type": "market", "custom_feature_enabled": true}

This enables:
  - best_bid_ask: dedicated event when best bid/ask changes (cleaner than
    parsing price_change for best prices)
  - new_market: could help detect new BTC 15-min contracts during rolls
    without REST polling
  - market_resolved: know immediately when current window resolves

── 5C. PM keepalive interval ──────────────────────────────────────────────────

Current:  PM_PING_INTERVAL = 8s
Docs say: every 10s

8s is fine (more frequent than required), but document that 10s is the
threshold. Connection drops if no PING received within ~10s.

8. RISKS & MITIGATIONS
================================================================================

Risk: update_subscription may not work as expected for rapid ticker changes
  Mitigation: Keep full reconnect as fallback, log subscription responses,
  use list_subscriptions to verify after swap

Risk: PM subscribe/unsubscribe has no server acknowledgment (silent success)
  Mitigation: Set a timer after swap — if no data arrives within 5s for new
  tokens, fall back to full reconnect

Risk: Kalshi SIDs may not persist across certain server-side events
  Mitigation: If recv loop gets unexpected errors after swap, trigger reconnect

Risk: fill channel may have high volume on busy markets
  Mitigation: Subscribe without market_ticker initially, scope to active
  ticker if volume is too high

Risk: RTDS connection adds a 3rd (or 4th) WS to manage
  Mitigation: RTDS is independent and non-critical — if it disconnects,
  BTC watcher still works, just no spot price display. Treat as best-effort.

Risk: PM user channel auth requires API creds that may not be configured
  Mitigation: Make PM fill tracking optional — only start user WS if
  POLYMARKET_API_KEY/SECRET/PASSPHRASE are all set in .env

Risk: Chainlink vs Binance price discrepancy confuses users
  Mitigation: Default to Chainlink only (matches PM resolution source).
  Offer Binance as optional comparison if requested.
